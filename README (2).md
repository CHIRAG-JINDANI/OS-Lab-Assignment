## 📌 Program 1 — Observing Variable Addresses Across Multiple Threads

### Objective
Demonstrate how local and global variable addresses differ across threads using `pthread`.

### Concepts
- Each thread gets its own **private stack** in virtual memory
- **Global variables** live in the shared data segment — same address for all threads
- Linux allocates **~8MB** per thread stack by default
- **ASLR** (Address Space Layout Randomization) randomizes base addresses on each run

### Compilation & Execution

```bash
cd /root/os-lab/Program1_Threads
gcc -o thread_addresses thread_addresses.c -lpthread
./thread_addresses | tee output.txt
```

### Key Observations

| Observation | Explanation |
|---|---|
| Local variable addresses differ per thread | Each thread has its own stack segment |
| Global variable address is identical in all threads | Global lives in shared data segment |
| Stack addresses spaced ~8MB apart | Default Linux thread stack size |
| Addresses differ between two runs | ASLR randomizes memory layout each execution |
| Thread execution order is non-deterministic | OS scheduler decides order, not programmer |

## 📌 Program 2 — Observing System Resource Utilization Using `top`

### Objective
Monitor CPU and memory usage in real time using `top` before, during, and after compiling and running a CPU-intensive program.

### Compilation & Execution

```bash
cd /root/os-lab/Program2_Top
gcc -O0 -o sample_program sample_program.c
./sample_program &
```

### Monitoring with top

```bash
# Real-time monitoring
top

# Batch mode snapshot (saved to file)
top -b -n 3 > output.txt
```

### Results Summary

| State | Free Memory | Top CPU Process | Top MEM Process |
|---|---|---|---|
| **Before** | 2553.5 MiB | `top` (~0%) | bash (~0.1%) |
| **During compile** | 2549.7 MiB | `gcc` (active) | bash (~0.1%) |
| **During run** | ~2450 MiB | `sample_program` (100%) | `sample_program` (2.6%, ~101MB) |

### Key Observations
- `gcc` spawns `collect2` and `ld` as sub-processes during compilation
- `sample_program` consumed **100% CPU** during matrix multiplication
- **50MB malloc** was clearly visible as increased RES in `top`
- Free memory dropped by ~3.8 MiB during compilation

## 📌 Program 3 — Exploring strace for System Call Tracing

### Objective
Trace system calls of executing processes using `strace` and identify shell built-in commands that make no `execve()` call.

### Compilation & Execution

```bash
cd /root/os-lab/Program3_Strace
gcc -o strace_demo strace_demo.c
```

### strace Experiments

```bash
# Filtered — show only file-related syscalls
strace -e openat,read,write,close ./strace_demo 2>&1 | tee strace_output.txt

# Summary table of all syscalls
strace -c ./strace_demo 2>&1 | tee strace_summary.txt

# Shell built-in — no execve() call
strace -e execve bash -c "cd /tmp" 2>&1

# Bash read/write calls
strace -e read,write bash -c 'echo "test"' 2>&1
```

### Key System Calls Observed

| System Call | Triggered By | Description |
|---|---|---|
| `execve()` | External commands only | Launches a new process |
| `openat()` | `fopen()` | Opens a file |
| `read()` | `fgets()` | Reads from file/stdin |
| `write()` | `printf()`, `fprintf()` | Writes to file/stdout |
| `close()` | `fclose()` | Closes file descriptor |
| `mmap()` | Program startup | Maps memory regions |
| `brk()` | `malloc()` | Adjusts heap boundary |
| `exit_group()` | `return 0` | Terminates process |

### Key Observations
- Shell built-in `cd` makes **no `execve()` call** — it runs entirely inside bash
- External commands like `ls`, `echo` trigger `execve()` to spawn a new process
- Even a simple `printf()` triggers multiple `write()` syscalls to stdout
- Program startup alone triggers 50+ syscalls before `main()` even runs (loading libc, mmap, etc.)

## 📌 Program 4 — Debugging a C Program with GDB

### Objective
Compile a C program with debug symbols and use GDB to trace variable values, memory addresses, function calls, and pointer manipulation.

### Features of the Program
- Multiple loops (`for`, `while`)
- File I/O with explicit `fflush()` buffering
- Functions: `find_largest()`, `swap()`, `sum_array()`
- Pointer manipulation and dynamic memory allocation

### Compilation in Debug Mode

```bash
cd /root/os-lab/Program4_GDB
gcc -g -Wall -o debug_program debug_program.c
./debug_program
```

### GDB Session

```bash
# Launch GDB
gdb ./debug_program
```

```gdb
# Set breakpoints
break main
break find_largest
break swap

# Run program
run

# Inspect variables
print a
print b
print &a
print &b
print *ptr_a

# Step through
next
next

# Inspect array memory
print arr
x/5d &arr[0]

# Watch variable change
watch count

# View call stack
backtrace

# Continue and quit
continue
quit
```

### Saved GDB Session

```bash
# Auto-save full GDB session to file
gdb -batch \
  -ex "break main" \
  -ex "run" \
  -ex "print a" \
  -ex "print b" \
  -ex "print &a" \
  -ex "next" \
  -ex "continue" \
  -ex "quit" \
  ./debug_program 2>&1 | tee gdb_session.txt
```

### Key Observations

| Observation | Explanation |
|---|---|
| Local variables have stack addresses (`0x7fff...`) | Variables live on the function's stack frame |
| Array elements are contiguous in memory | `&arr[1] - &arr[0] = 4 bytes` (sizeof int) |
| Pointer `*ptr_a` gives same value as `a` | Pointer correctly dereferences to variable |
| After `swap()`, values of a and b are exchanged | Pass-by-pointer correctly modifies originals |
| `watch count` pauses on every decrement | GDB hardware watchpoint detects memory writes |
| `backtrace` shows call chain | Confirms `main → find_largest` call hierarchy |
| `fflush(fp)` forces buffer to disk immediately | Without it, data may stay in kernel buffer |
